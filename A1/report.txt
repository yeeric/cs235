PROBLEM DESCRIPTION
-------------------
This assignment compares two algorithms for the membership problem:
- Algorithm A: Store set S in an unsorted list, use linear search (O(n) per search)
- Algorithm B: Sort the list once (O(n log n)), then use binary search (O(log n) per search)

The goal is to find k*, the minimum number of searches where Algorithm B becomes
preferable to Algorithm A, and analyze how the ratio k*/n changes as n increases.


IMPLEMENTATION DETAILS
----------------------
- Algorithm A: Simple linear search iterating through the list
- Algorithm B: Merge sort (O(n log n)) followed by binary search (O(log n))
- Both sort and binary search were implemented from scratch as required
- Timing measured using Python's time.perf_counter() for high precision
- Each experiment repeated 3 times and averaged to account for system variation
- Target values: half selected from S, half guaranteed not to be in S

ANALYSIS AND FINDINGS
---------------------

1. TREND OBSERVATION:
   As n increases, the ratio k*/n DECREASES.
   
   - At n=1000:  k*/n = 0.1390 (need ~14% of n searches for B to be better)
   - At n=2000:  k*/n = 0.0755 (need ~7.5% of n searches)
   - At n=5000:  k*/n = 0.0350 (need ~3.5% of n searches)
   - At n=10000: k*/n = 0.0190 (need ~1.9% of n searches)

2. EXPLANATION:
   This trend makes sense from a theoretical perspective:
   
   - Algorithm A's total time for k searches: O(k * n)
   - Algorithm B's total time: O(n log n) for sorting + O(k log n) for searches
   
   Algorithm B is preferable when:
       k * n > n * log(n) + k * log(n)
       k * n - k * log(n) > n * log(n)
       k * (n - log(n)) > n * log(n)
       k > (n * log(n)) / (n - log(n))
   
   For large n, (n - log n) ≈ n, so:
       k* ≈ log(n)
   
   Therefore:
       k*/n ≈ log(n)/n
   
   As n increases, log(n)/n DECREASES because n grows faster than log(n).
   This matches our experimental observations perfectly.

3. PRACTICAL IMPLICATIONS:
   - For small datasets, many searches are needed before sorting pays off
   - For large datasets, the sorting overhead is quickly amortized
   - The absolute value of k* grows slowly (roughly logarithmically with n)
   - The ratio k*/n shrinks as datasets get larger

4. WHY k* INCREASES SLOWLY:
   - k* values: 139, 151, 175, 190 for n = 1000, 2000, 5000, 10000
   - k* grows approximately as O(log n), not O(n)
   - This is because the sorting cost (n log n) only needs to be offset by
     the savings per search (n - log n ≈ n for each search)


CONCLUSION
----------
The experiments confirm that Algorithm B (sort + binary search) becomes increasingly
advantageous as the dataset size grows. While the absolute number of searches k*
needed for Algorithm B to become preferable does increase with n, it increases
very slowly (logarithmically). The ratio k*/n decreases as n increases, meaning
that for larger datasets, a smaller PROPORTION of searches is needed to justify
the upfront sorting cost.

This demonstrates the fundamental importance of choosing appropriate data structures
and algorithms based on the expected use case. For applications with many searches
on the same dataset, the O(n log n) sorting investment quickly pays for itself
through O(log n) search times instead of O(n).
